## 技術方案提案

### 1. 全体アーキテクチャ概要

要件を満たすため、以下の2段階のパイプラインを提案します。

- **第1段階（高速スクリーニング）** ：YOLO-Worldを用いて、クエリ画像全体から対象クラス候補を高速にスキャンします。
- **第2段階（精密照合）** ：検出された候補領域に対し、事前計算したクラスプロトタイプ（参考画像埋め込み）との類似度を再評価し、誤検出を排除します。

YOLO-WorldはYOLOアーキテクチャのリアルタイム性とCLIPのオープンボキャブラリ認識能力を組み合わせており、精度を落とさずに高効率な推論が可能です。このハイブリッド構成により、速度と精度の両立が可能です。また、Training-freeの要件を満たし、頻繁なクラス追加にも柔軟に対応できます。

本提案の要点は以下の通りです：

- **既存モデルをトレーニング不要で利用**：YOLO-WorldやGrounding DINOなどのゼロショット物体検出モデルは、追加学習なしで任意のクラス名による検出が可能です。
- **画像プロンプト（参考写真）の活用**：参考写真が1〜5枚提供されるという利点を活かし、これらの画像埋め込みをクラスプロトタイプとして活用します。このアプローチはSearchDet（CVPR 2025）の知見と一致しており、参照画像を用いることでテキストのみの場合より高い精度が得られます。
- **モジュール構成による柔軟性**：クラスの追加・削除は、対応するプロトタイプベクトルの追加・削除のみで対応でき、モデルの再学習は一切不要です。

### 2. モジュール構成と処理フロー

本ソリューションは以下の5つのモジュールで構成されます。

#### モジュール1：画像前処理部

- 入力：クエリ画像（JPEG/PNG形式、任意の解像度）
- 処理：YOLO-Worldの推論に適したサイズ（640×640）へのリサイズと正規化
- 出力：前処理済み画像テンソル

#### モジュール2：クラスプロトタイプ管理部

- **プロトタイプ生成（クラス登録時）**
    1. 各クラスの参考画像（1〜5枚）を読み込み、DINOv2で特徴ベクトルを抽出します。
    2. 抽出したベクトルをL2正規化します。
    3. 複数枚ある場合は平均プーリングまたは最大プーリングにより単一のプロトタイプベクトルに統合します。
    4. クラス名、カテゴリ情報とともに埋め込みベクトルデータベース（FAISSなど）に保存します。
- **プロトタイプ削除（クラス削除時）** ：該当クラスのベクトルをデータベースから削除します。

このFew-shotプロトタイプ構築はHMKM（Hierarchical Multimodal Knowledge Matching）などの研究とも整合します。

#### モジュール3：YOLO-Worldによる高速検出部

- 入力：前処理済みクエリ画像、対象クラス名リスト（テキストプロンプト）
- 処理：YOLO-World（例：`yolo-world-l`）で推論を実行し、バウンディングボックス候補を生成します。
- 出力：バウンディングボックス座標と信頼度スコアのリスト

#### モジュール4：プロトタイプ照合・再評価部

- 入力：YOLO-Worldが生成したバウンディングボックス候補、各クラスのプロトタイプベクトル
- 処理：
    1. 各バウンディングボックス領域をDINOv2でベクトル化し、L2正規化します。
    2. 対象クラスのプロトタイプベクトルとのコサイン類似度を計算します。
    3. 類似度が閾値（例：0.7）を上回った場合のみ最終検出として採用します。
    4. 必要に応じてNMS（Non-Maximum Suppression）で重複ボックスを統合します。
- 出力：最終的なバウンディングボックス座標と信頼度スコア

#### モジュール5：結果出力部

- 入力：最終検出ボックスとスコア
- 処理：JSON形式で結果を整形
- 出力：以下の形式で出力します。

```json
{
  "image_id": "query_001.jpg",
  "detections": [
    {"class_name": "Pokemon Treecko", "bbox": [x1, y1, x2, y2], "confidence": 0.92, "category": "toy"},
    {"class_name": "Coca-Cola 500ml", "bbox": [x1, y1, x2, y2], "confidence": 0.88, "category": "beverage"}
  ]
}
```

### 3. 精度向上のための補完手法

基本構成に加え、以下の補完手法を導入することで精度をさらに向上できます。

#### プロトタイプ構築の工夫

HMKMの知見を活かし、各クラスの物体の属性（例：「緑色」「円筒形」「ロゴあり」など）をプロトタイプ知識として事前定義し、類似クラスとの識別に活用します。特にカテゴリ情報（`category='toy'`など）が利用可能な場合、同じカテゴリ内で類似度の閾値を動的に調整することで誤検出を抑制できます。

#### 代替モデルの選択肢

要件の「精度優先」を踏まえると、以下の選択肢も状況に応じて有効です：

- **Grounding DINO**：Transformerベースで高精度なローカリゼーションに優れており、重要度の高い検査タスクに適しています。ただし推論速度はYOLO-Worldより遅くなります。
- **OWLv2**：Meta AIの開発で、Webスケールの学習により希少クラスの汎化に優れています。画像とテキストの両方をプロンプトとして利用できるため、参考写真を直接クエリとして使用することも可能です。
- **SearchDet**：参考画像セットからクエリ埋め込みを生成し、入力画像内の対象物体を検出する手法で、トレーニング不要でありながらGrounding DINOを大幅に上回る精度（ODinWで+16.81% mAP）を報告しています。複数枚の参考写真を効果的に活用したい場合に有力な選択肢となります。

T4 GPU 1枚の環境では、精度優先のユースケースにおいてYOLO-World LまたはGrounding DINOのいずれも十分に動作します。タスクの性質（高精度が絶対条件か、ある程度の速度も両立したいか）に応じて選択することを推奨します。

### 4. T4 GPUでの推論時間見積もり

YOLO-World L（640×640入力）を前提とした場合：

- **640×480入力**：約0.8〜1.2秒（画像リサイズ + 特徴抽出 + プロトタイプ照合を含む全体パイプライン）
- **1600×1200入力**：約1.5〜2.5秒（リサイズ処理のオーバーヘッド増加を含む）

実際の処理時間はクエリ画像中のオブジェクト数や検出クラス数に依存します。YOLO-WorldはLVISデータセットでV100上で52.0 FPSを達成しており、T4でも十分な性能が期待できます。バッチ処理を導入すれば、複数画像の処理時間をさらに短縮できます。

精度をさらに追求してGrounding DINOを採用した場合、推論時間は約3〜5倍に増加します（640×480で約3〜5秒、1600×1200で約6〜10秒）。どちらを選ぶかは、プロジェクトの精度要件と処理速度要件のバランスに依存します。


## 付随質問への回答

### 質問1：正面と背面が異なるケースへの対応

あるクラスの物体の正面と背面が全く異なり、かつ背面が他クラスの背面と同じ場合、以下の対策が有効です。

#### 対応策
- **多視点プロトタイプの活用**：各クラスの参考写真を単一のベクトルに統合せず、各視点ごとに複数のプロトタイプとして保持します。検出時には、クエリ領域が**いずれかの視点プロトタイプと一致すれば**同一クラスと判定します。
- **階層的クラス定義の導入**：カテゴリ情報（`category='toy'`）を活用し、まずカテゴリレベルで判定し、その後に細粒度クラスを判定する2段階方式を採用します。これにより背面の類似による誤分類を抑制できます。
- **類似度閾値の動的調整**：同じカテゴリ内で類似度が拮抗した場合、正面視点の類似度をより重視する重み付けを行います。

#### 参考写真の準備指針
1. **最低限の構成**：正面、背面、側面（右または左）の最低3枚を推奨します。
2. **理想的な構成（最大5枚）** ：正面、背面、右側面、左側面、上面（または斜め上からの俯瞰）の5枚を推奨します。これにより物体の全方向からの特徴をカバーできます。
3. **データ品質**：背景は可能な限り単一色（白または無地）とし、物体が明確に映っているものを使用してください。
4. **補足情報の活用**：カテゴリ情報を活用し、同じカテゴリ内の類似物体との混同を防ぐための属性情報（色、素材、特徴的な形状など）をメタデータとして付与することを推奨します。

### 質問2：推論時間

上記「T4 GPUでの推論時間見積もり」で回答済みです。再掲すると：

- **640×480入力**：約0.8〜1.2秒（YOLO-Worldベース）/ 約3〜5秒（Grounding DINOベース）
- **1600×1200入力**：約1.5〜2.5秒（YOLO-Worldベース）/ 約6〜10秒（Grounding DINOベース）

これは全体パイプライン（前処理、モデル推論、プロトタイプ照合、後処理を含む）の時間です。実際の処理時間は画像内のオブジェクト数や検出対象クラス数に応じて変動します。


## 補足事項

### 実行環境と前提条件
- **GPU**：NVIDIA Tesla T4（16GB VRAM）1枚
- **フレームワーク**：PyTorch 2.x、Ultralytics、Transformers
- **ベクトルデータベース**：FAISS（CPU動作、メモリ使用量はクラス数×ベクトル次元数に依存）
- **前提**：YOLO-WorldやGrounding DINOのモデル重み（数百MB〜数GB）は事前にダウンロードし、ディスク上にキャッシュ済みであること

### 注意点
- YOLO-Worldはオープンボキャブラリ検出が可能ですが、学習データに含まれない極めて特殊な物体（例：特定の限定版フィギュア）では精度が低下する可能性があります。その場合、参考写真によるプロトタイプ照合が精度を補完します。
- 参考写真の品質が低い（低解像度、不適切な照明、物体の一部しか写っていないなど）場合、プロトタイプの品質が低下し、誤検出や未検出の原因となります。可能な限り高品質な画像の用意を推奨します。
- 数千クラスのプロトタイプベクトルを全てメモリ上に保持する場合、DINOv2-base（768次元）で約数MB〜数十MB程度のメモリが必要です。これはT4のVRAMとは別にシステムRAM上で管理されるため、GPUメモリを圧迫しません。
